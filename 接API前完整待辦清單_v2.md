## 五、🆕 安全防護機制（接API前必做，本輪新增）

### 背景
經討論發現，現有免費額度機制（風水問答3條、求簽每日1次）**純粹依賴前端localStorage**，沒有任何後端驗證。現階段因答案為靜態mock，被繞過的直接財務損失為零；但**一旦接入真實Anthropic API，所有繞過漏洞都會直接轉化為真實金錢損失**。

### 已確認嘅漏洞
| 漏洞 | 難度 | 影響 |
|---|---|---|
| 清除瀏覽器Cache/LocalStorage | 極易 | 完全重置額度 |
| 無痕模式（Incognito） | 極易 | 每次新視窗即重置 |
| Devtools Console手動清除 | 極易 | 即時重置 |
| oracle.html測試用重置按鈕 | 極易（一鍵） | 直接重置，比上面三者更直接 |
| Gmail「+號」trick開「無限」email帳號 | 中等 | 如未來用email做帳號限制，呢個漏洞依然成立 |
| Prompt injection（誘導AI輸出超長內容或離題內容） | 中等 | 即使用戶已付費，仍可能令單次API成本暴增超出預期 |

### 必須建立嘅3層防禦（接API前必做）

**第一層：System Prompt限制（基礎，但不足夠）**
- 明確指示AI只回答指定範疇問題（風水/命理），其他主題禮貌拒絕
- 單靠呢一層唔可靠，必須配合下面兩層

**第二層：後端身份驗證+配額（核心，需要Firebase或同等後端）**
- API呼叫計數移至後端資料庫（Firebase Firestore），唔可以純靠前端localStorage
- 帳號登入後配額綁定UID，登出用戶只能看mock答案
- 補充訊號：device fingerprint + IP限速，提高重置難度（即使用戶登出/換瀏覽器都有阻力）
- 月費「每月50條」軟上限，亦必須喺後端強制執行，唔可以前端判斷（FEATURES.md已有相關記錄）

**第三層：技術上限制單次成本（直接防止燒爆token）**
- API call時設定`max_tokens`硬上限（例如風水問答固定輸出上限300 tokens）
- 後端檢查用戶輸入長度，超過上限（例如100字）直接拒絕，唔送去API
- 呢一層保護緊**所有**用戶（免費同付費都要有），唔淨係「免費用戶」專屬，因為已付費用戶都可能透過prompt injection令單次成本暴增

### 即時行動（接API前嗰一刻，必須完成）
1. 移除/隱藏oracle.html測試用重置按鈕（用戶已確認，攞到API後執行）
2. 第二層後端配額系統必須喺Firebase接入嗰一個sprint一次過做好，**唔可以「先接API、後補安全」**，順序必須係：先有後端配額驗證，先開放真實API call
